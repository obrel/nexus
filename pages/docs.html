Nexus Documentation

Welcome to the Nexus developer documentation. Nexus is a high-concurrency, distributed messaging service designed to scale to millions of concurrent WebSocket connections.

CQRS-Inspired Architecture: Nexus separates message ingestion (HTTP Ingress) from message delivery (WebSocket Egress) for maximum scalability.

Key Components

  • Ingress Tier: Stateless HTTP endpoints for command processing.
  • Egress Tier: Stateful WebSocket connections for real-time delivery.
  • Message Bus: NATS cluster for internal low-latency communication.

Overview

Nexus is designed for extreme horizontal scalability. By decoupling ingestion from delivery, each tier can be scaled independently based on load patterns.

Architecture

graph TD Client[Client/Publisher] -->|HTTP POST| LB[Load Balancer] LB --> Ingress1[Ingress Node A] LB --> Ingress2[Ingress Node B] Ingress1 -->|Publish| NATS{NATS Cluster} Ingress2 -->|Publish| NATS Ingress1 -->|Async Write| MySQL[(MySQL Cluster)] Ingress2 -->|Async Write| MySQL NATS -->|Subscribe| Egress1[Egress Node X] NATS -->|Subscribe| Egress2[Egress Node Y] Egress1 -->|WebSocket| UserA[User A] Egress2 -->|WebSocket| UserB[User B] Egress1 <--> Redis[(Redis Metadata)] Egress2 <--> Redis[(Redis Metadata)]

Message Flow

High-level sequence of a group message being published and delivered.

sequenceDiagram participant A as Sender (Client A) participant I as Ingress (HTTP) participant N as NATS Bus participant E as Egress (WS) participant B as Recipient (Client B) A->>I: 1. POST /v1/messages/send I->>A: 2. 202 Accepted (message_id) Note over I: Outbox relay publishes to NATS I->>N: PUB nexus.{app_id}.v1.grp.{shard}.{group_id} N->>E: 3. MSG to Egress Node E->>B: 4. WS event: MSG_NEW B->>I: 5. POST /v1/messages/ack (status: received) I->>N: PUB MSG_STATUS to sender's subject N->>E: 6. Relay to Sender's Egress E->>A: 7. WS event: MSG_STATUS (received)

Presence Management

Nexus tracks user online status via periodic heartbeats over the WebSocket connection.

sequenceDiagram participant C as Client (WS) participant E as Egress Node participant R as Redis participant N as NATS Bus C->>E: Connection Established E->>R: SETEX user:presence:{user_id} 60 online E->>N: PUB presence.status.{user_id} online loop Every 30s (Heartbeat) C->>E: Ping / Heartbeat E->>R: EXPIRE user:presence:{user_id} 60 end C->>E: Connection Closed (Clean) E->>R: DEL user:presence:{user_id} E->>N: PUB presence.status.{user_id} offline

Installation

To get started with Nexus, ensure you have the following prerequisites installed:

  • Go 1.24+
  • MySQL 8.0+
  • Redis 7.0+
  • NATS Server

Clone the Repository

git clone https://github.com/obrel/nexus.git
cd nexus

Setup Configuration

cp config.yaml.example config.yaml

Database Migrations

Nexus uses golang-migrate to manage database schema versions. The migrate subcommand provides several tools for state management.

Run Pending Migrations

Apply all pending .up.sql files found in db/migrations/.

go run ./cmd/nexus migrate up

Rollback Last Migration

Rollback the most recent migration step.

go run ./cmd/nexus migrate down

Check Version

Show the current migration version and check for dirty states.

go run ./cmd/nexus migrate version

Note: If a migration fails and leaves the database in a "dirty" state, you can clear it using the force command: go run ./cmd/nexus migrate force <version>.

Configuration

Nexus uses Viper for configuration, supporting config.yaml and environment variables. Below is the complete reference of available settings.

Application

Key Type Description
app.id string Unique identifier for this Nexus deployment.
app.env string development or production.

HTTP Ingress

Key Default Description
http.port 8080 Port for the HTTP Ingress server.
http.jwt_secret - Secret key for signing and verifying JWTs.
http.internal_api_key - Shared secret for internal service-to-service calls (required in production, min 32 chars).
http.internal_routes true Enable or disable all /internal/* endpoints. Set to false to prevent any service-to-service access.
http.max_body_size 1MB Maximum size for request bodies.
http.max_header_bytes 1MB Maximum size for HTTP headers.

WebSocket Egress

Key Default Description
ws.port 8081 Port for the WebSocket server.
ws.ping_period 30s Interval for sending pings to clients.
ws.pong_wait 90s Time allowed to read the next pong from a client.
ws.write_wait 10s Time allowed to write a message to a client.
ws.presence_ttl 60s Time before a user is marked offline in Redis.

Infrastructure

Configuring MySQL, Redis, and NATS connections.

Key Default Description
mysql.max_open_conns 100 Maximum number of open database connections.
mysql.max_idle_conns 25 Maximum number of idle database connections.
nats.sub_pending_msgs_limit 65536 Maximum pending messages for a NATS subscription.
nats.sub_pending_bytes_limit 64MB Maximum pending bytes for a NATS subscription.

Background Worker

Key Default Description
worker.batch_size 50 Number of outbox entries to process per poll.
worker.poll_interval 1000ms Interval between outbox polling.

Running Nexus

Nexus is a single binary with multiple subcommands. In a production environment, you would typically scale each tier independently.

Start HTTP Ingress

go run ./cmd/nexus http

Start WebSocket Egress

go run ./cmd/nexus ws

Start Background Worker

The worker handles the transactional outbox relay to NATS.

go run ./cmd/nexus worker

Using Docker Compose

For a quick spin-up including all dependencies and automatic database migrations:

docker compose up --build

Authentication

Nexus uses JWT (HS256) for authenticating all API requests and WebSocket connections. Tokens are passed via the Authorization: Bearer <token> header.

Nexus does not provide user registration or login flows. Developers implement their own auth service and use the internal API to sync users into Nexus.

JWT Format

Nexus expects the following JWT claims:

Claim Type Description
sub string User ID (returned by the register endpoint).
tid string Tenant / App ID.
exp number Expiration time (Unix timestamp).
iat number Issued at (Unix timestamp).

Generating a JWT (Self-Managed Auth)

If you manage authentication yourself instead of using the internal registration API, you can generate JWTs directly using the same http.jwt_secret configured in Nexus. The user must already exist in the Nexus users table.

Go:

import (
    "time"
    "github.com/golang-jwt/jwt/v5"
)

func GenerateNexusJWT(jwtSecret, appID, userID string) (string, error) {
    claims := jwt.MapClaims{
        "sub": userID,
        "tid": appID,
        "iat": time.Now().Unix(),
        "exp": time.Now().Add(24 * time.Hour).Unix(),
    }
    token := jwt.NewWithClaims(jwt.SigningMethodHS256, claims)
    return token.SignedString([]byte(jwtSecret))
}

Node.js:

const jwt = require("jsonwebtoken");

function generateNexusJWT(jwtSecret, appID, userID) {
  return jwt.sign(
    { sub: userID, tid: appID },
    jwtSecret,
    { expiresIn: "24h" }
  );
}

Python:

import jwt, time

def generate_nexus_jwt(jwt_secret: str, app_id: str, user_id: str) -> str:
    now = int(time.time())
    payload = {
        "sub": user_id,
        "tid": app_id,
        "iat": now,
        "exp": now + 86400,  # 24 hours
    }
    return jwt.encode(payload, jwt_secret, algorithm="HS256")

Important: The jwt_secret must match the value configured in the Nexus server (http.jwt_secret). In production, use a strong secret of at least 32 characters.

Messages API

All message operations are performed via the Ingress HTTP API. The same endpoint handles both group and private messages. For messages originating from an external service, see the internal API.

Send Group Message

The sender must be a member of the target group. Non-members receive 403 Forbidden with error code not_a_member.

POST /v1/messages/send
{
  "group_id": "nexus_dev",
  "content": { "type": "text", "text": "Hello World!" }
}

Send Private Message

Use user_id instead of group_id to send a direct message to a specific user. No membership check is required for private messages.

POST /v1/messages/send
{
  "user_id": "420158066212864",
  "content": { "type": "text", "text": "Hey, are you available?" }
}

Note: group_id and user_id are mutually exclusive — exactly one must be provided. Group messages require membership; private messages are delivered via the recipient's private WebSocket subscription.

Edit Message

Edit the content of a message you sent. Only the original sender can edit their own messages.

POST /v1/messages/edit
{
  "message_id": "420158066278400",
  "content": { "type": "text", "text": "Updated message content" }
}

// Response (200 OK)
{
  "success": true,
  "data": { "message_id": "420158066278400" }
}

Delete Message

Soft-delete a message you sent. Only the original sender can delete their own messages. A MSG_UPDATED event with "action": "deleted" is broadcast to recipients.

POST /v1/messages/delete
{
  "message_id": "420158066278400"
}

// Response (200 OK)
{
  "success": true,
  "data": { "message_id": "420158066278400" }
}

Acknowledge Message

Signal that a message has been received or read. Works for both group and private messages. Valid statuses: received, read.

POST /v1/messages/ack
{
  "message_id": "420158066278400",
  "status": "read"
}

// Response (200 OK)
{
  "success": true,
  "data": {}
}

For service-to-service group management, see the internal API.

Create Group

Retrieve group details including the creator. Only group members can view group details.

{
  "success": true,
  "data": {
    "group_id": "nexus_dev",

    "description": "Main development channel",
    "created_by": "420158066212864",
    "created_at": "2026-03-22T10:00:00Z"
  }
}

Update Group

Update a group's name and/or description. Only group members can update a group. Non-members receive 403 Forbidden with error code not_a_member.

POST /v1/groups/update
{
  "group_id": "nexus_dev",
  "name": "Nexus Core",
  "description": "Main development channel"
}

Join Group

Joining a group automatically triggers a WebSocket subscription for the user. A USR_JOINED event is broadcast to existing group members.

POST /v1/groups/join
{
  "group_id": "nexus_dev"
}

Leave Group

Leaving a group removes the user's membership and unsubscribes them from the group's shard if they have no other groups on that shard.

POST /v1/groups/leave
{
  "group_id": "nexus_dev"
}

Internal API (Service-to-Service)

Nexus exposes a set of internal endpoints under /internal/v1/ for service-to-service communication. These allow an external service to perform operations on behalf of a user without requiring a JWT token — the acting user's identity is supplied explicitly in the request body instead.

Configuration

Internal routes are controlled by two settings in config.yaml:

Setting Default Description
http.internal_routes true Master switch. Set to false to disable all /internal/* endpoints entirely.
http.internal_api_key — Shared secret that callers must supply via the X-Internal-Key header. Required in production (min 32 chars). Routes are not registered if this is empty.

Note: Internal routes are only registered when both http.internal_routes: true and a non-empty http.internal_api_key are configured. Every request to an internal endpoint must include the header X-Internal-Key: <your-key>.

User Registration

Syncs a user from an external auth service into Nexus. Creates the user if they don't exist, or returns the existing record, and issues a fresh JWT. name and app_id are optional.

POST /internal/v1/auth/register
X-Internal-Key: your-internal-api-key

{
  "email": "user@example.com",
  "name": "John Doe",
  "app_id": "myapp"
}

// Response (200 OK)
{
  "success": true,
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIs...",
    "user_id": "420158066212864"
  }
}

Messages

Internal counterparts for the full message lifecycle. All endpoints require app_id and the acting user's ID in the request body.

Send Message

Send a group or private message on behalf of a user. group_id and user_id are mutually exclusive.

POST /internal/v1/messages/send
X-Internal-Key: your-internal-api-key

{

{

}

History API

Retrieve message history for a group or private conversation.

Group History

GET /v1/history?group_id=nexus_dev&since_id=0&limit=25

Private History

Use user_id to fetch the conversation history between you and another user.

GET /v1/history?user_id=420158066212864&since_id=0&limit=25

Note: group_id and user_id are mutually exclusive. Group history requires group membership. Private history is accessible by both participants.

WebSocket Integration

Connection to the Egress tier is required to receive real-time updates. The Egress tier is responsible for pushing events to clients.

Connection URL

ws://localhost:8081/ws

Note: Authentication is typically handled via subprotocols or query parameters if headers are not supported by the client library during the handshake.

Downstream Events

Nexus pushes events to connected clients in a standard JSON envelope:

{
  "event": "EVENT_TYPE",
  "data": { ... }
}

MSG_NEW

A new message has been posted to a group or received as a private message.

{
  "event": "MSG_NEW",
  "data": {
    "message_id": "420158066278400",
    "recipient_type": "group",
    "recipient_id": "nexus_dev",
    "sender_type": "user",
    "sender_id": "420158066212864",
    "content": { "type": "text", "text": "Hello everyone!" }
  }
}

Note: For private messages, recipient_type is "user" and recipient_id is the target user's ID. The content field is a JSON object; currently only "type": "text" is supported.

MSG_UPDATED

A message has been edited or deleted. The action field indicates which operation occurred.

// Edited
{
  "event": "MSG_UPDATED",
  "data": {
    "message_id": "420158066278400",
    "recipient_type": "group",
    "recipient_id": "nexus_dev",
    "content": { "type": "text", "text": "Hello everyone! (edited)" },
    "action": "edited"
  }
}

// Deleted
{
  "event": "MSG_UPDATED",
  "data": {
    "message_id": "420158066278400",
    "recipient_type": "group",
    "recipient_id": "nexus_dev",
    "action": "deleted"
  }
}

MSG_STATUS

Receipt or read status update for a message. Delivered to the original sender's private WebSocket subscription.

{
  "event": "MSG_STATUS",
  "data": {
    "message_id": "420158066278400",
    "status": "read",
    "user_id": "420158066212864"
  }
}

Note: user_id is the user who acknowledged the message. status is either "received" or "read".

USR_JOINED

A new user joined the group. Broadcast to all current group members.

{
  "event": "USR_JOINED",
  "data": {
    "group_id": "nexus_dev",
    "user_id": "420158066212864"
  }
}

TYP_INDICATOR

Real-time typing status change. Broadcast to the group or delivered to the private recipient.

// Group typing
{
  "event": "TYP_INDICATOR",
  "data": {
    "recipient_type": "group",
    "recipient_id": "nexus_dev",
    "user_id": "420158066212864",
    "typing": true
  }
}

// Private typing
{
  "event": "TYP_INDICATOR",
  "data": {
    "recipient_type": "user",
    "recipient_id": "420158066212865",
    "user_id": "420158066212864",
    "typing": false
  }
}

Upstream Indicators

Typing indicators are the only messages sent upstream via the WebSocket connection. Use group_id for group typing or user_id for private typing.

Start Typing (Group)

{
  "request": "TYP_START",
  "data": { "group_id": "nexus_dev" }
}

Start Typing (Private)

{
  "request": "TYP_START",
  "data": { "user_id": "420158066212864" }
}

Stop Typing

{
  "request": "TYP_STOP",
  "data": { "group_id": "nexus_dev" }
}
// or
{
  "request": "TYP_STOP",
  "data": { "user_id": "420158066212864" }
}
